Описание задания
Используя полученные знания, вам предстоит реализовать сервис рекомендаций фильмов и сериалов, используя как контентные признаки, так и данные о лайках/дизлайках пользователей.

  

Описание рекомендательной системы
Рекомендательная система состоит из трех основных компонент:

Recommendations service

Бекенд сервис, отвечающий за рекомендации конечным пользователям, имеющая следующие эндпоинты (роуты/ручки):

/healthcheck возвращает статус сервиса. Для грейдера необходимо, чтобы возвращался код ответа 200 (OK)
/cleanup производит сброс окружения перед новым запуском грейдера (при каждом запуске идентификаторы новые и нужно очистить все возможные кэширования для корректной проверки)
/add_items добавляет в систему новые объекты рекомендации (поле item_ids содержит список идентификаторов объектов, а genres – список списков жанров для соответствующих объектов в item_ids)
/recs/{user_id} возвращает список рекомендаций в виде списка объектов для пользователя user_id
При проверке запросы будут идти на 5001 порт

Event collector

Сервис сбора и обработки реакции пользователей (лайки/дизлайки), обрабатывающий эндпоинт /interact. В бейзлайне просто сохраняет полученные события в RabbitMQ для дальнейшей обработки в сервисе Regular pipeline

При проверке запросы будут идти на 5000 порт

Regular pipeline

Аналог планировщика задач Airflow/Luigi, в случае данного задания просто скрипт с циклическим обучением алгоритмов. Рекомендуется сохранять список рекомендаций непосредственно в Redis, а вектора в Quadrant для быстрой обработки запросов пользователей.


Оценивание
Когда вы отправите IP-адрес вашего сервиса, на котором запущен сервис рекомендаций и event-collector, на него начнут поступать запросы GET для рекомендации и POST для сохранения фидбека от рекомендаций. Для каждого пользователя определен список релевантных объектов, которые он оценил с помощью событий like/dislike. При этом, для каждого пользователя будет поступать несколько запросов на рекомендацию, между которыми есть задержка (как короткая, так и долгая, чтобы обучить модель с персональными рекомендациями), чтобы учесть фидбек или отфильтровать уже показанные объекты. В один момент времени запрашиваются рекомендации максимум для 50 пользователей.

Оценка за задание состоит из нескольких составляющих (для каждого пункта финальная оценка выглядит как линейная функция от минимального значения до максимального):

(10 баллов) метрика качества рекомендаций precision (минимум 0.02, максимум 0.15)
(10 баллов) метрика качества рекомендаций ndcg (минимум 0.02, максимум 0.07)
(10 баллов) метрика разнообразия рекомендаций diversity (минимум 0.1, максимум 0.4)
(10 баллов) метрика покрытия каталога объектов coverage (минимум 0.1, максимум 0.8)
(бонусные 10 баллов) скорость обработки запросов пользователей (время ответа на запрос, минимум баллов – 800 мс, максимум – 80 мс)
Бонусные баллы работают только при достижении хотя бы 30 баллов за другие пункты

При подсчете метрик качества используется TOP_K = 10

Советы по решению
У вас есть несколько попыток заинтересовать пользователя, при этом один и тот же объект в новой сессии не будет учитываться как релевантный, поэтому нужно исследовать интересы
Не забывайте про проблемы feedback loop и cold start (как для пользователя, так и для объектов)
Тест длится долгое время, вы можете настроить на стороне сервера метрики в prometheus или mlflow, чтобы отслеживать, как работает ваше приложение в реальном времени
